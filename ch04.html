第 4 章
檢測入門

了解可觀測性這項實務背後的整體理論與起源之後，接下來我們要進入技術基礎與實作的具體細節。首先，我們要探討檢測的基本原理。

商業界有句老話大致是這麼說的：「你無法改善你不衡量的東西。」這個觀念套用在你所建置、具備高度可觀測性的系統上同樣成立。由於現代應用程式系統有太多不同的功能面向，實務上，我們必須收集數量日益龐大的資料，才能完全理解這些系統。

正如本書這個部分所述，產生並收集遙測資料是可觀測系統的基礎基石。只要有足夠數量的遙測資料可供分析，就有可能理解系統的任何一種狀態——無論它多麼新奇或詭異。然而，在開始討論資料分析之前，我們必須先深入了解「足夠數量的遙測資料」究竟代表什麼意思。讓我們一起來看看，要適當地檢測系統究竟需要做些什麼。

本章將介紹檢測的入門知識，並針對如何產生遙測資料以及該把遙測資料送往何處提出一些考量。身為負責確保應用程式可靠性的軟體工程師，這些考量必須成為你工作流程中不可或缺的一環。

我們也會簡短介紹所有軟體工程師在探討可觀測性時都必須了解的概念：OpenTelemetry 專案（一個開源且與廠商無關的檢測框架）、部署策略，以及自訂設定、成本與取樣等實務考量。本章著重於入門層級的資訊，但這些概念會在本書後續內容中進一步深入探討。

檢測基礎

軟體工程中的檢測是什麼？這是一種在應用程式中加入程式碼的實踐，讓程式在執行時能夠發出關於其運作狀況的資料。你可以利用這些資料來理解軟體行為、衡量性能，或是診斷執行期錯誤。

打個比方，現代可觀測系統就像是夜間搭機返家時，望向窗外，俯瞰繁華城市的燈火。成千上萬的燈光忽明忽滅，車流穿梭於錯綜複雜的道路網絡之間，這些複雜的活動模式雖然美麗，卻難以在缺乏適當視角的情況下加以理解。

正如城市規劃者需要交通感測器、人口監測器和公用事業計量器才能理解他們的城市如何運作，軟體工程師也需要檢測才能理解他們的應用程式在正式環境中的行為。本書其中一位作者是在為本書進行了幾天密集的規劃工作後想出這個比喻的。顯然那項工作讓他吃了不少苦頭，但我們認為這個比喻仍然站得住腳。

你可以把檢測想像成在應用程式這座龐大的城市各處安裝感測器與計量器，用來回報從資料庫查詢耗時多久、用戶實際使用哪些功能，到發生錯誤時究竟發生了什麼事等各種資訊。這些遙測資料成為可觀測性的基礎——也就是你理解並解釋系統可能進入的任何狀態的能力，無論該狀態多麼新奇或複雜。檢測越全面，你的理解就越細緻。

沒有適當的檢測，除錯正式環境問題就會變成一場證據不足的偵探工作。你或許知道用戶正經歷延遲緩慢的問題，但你不會知道究竟是哪些具體操作變慢了、影響了哪些用戶，又是在什麼條件下發生的。如果一開始沒有做好適當的檢測，你就會發現自己是在問題發生後才加入日誌陳述式，等待問題重現，並做出充滿邏輯跳躍的臆測，試圖找出難以捉摸的根本原因，以及更加難以捉摸的促成因素。

有了良好的檢測，你便能有系統地處理問題，順著資料的脈絡準確理解發生了什麼事以及原因為何。本書的這個部分將教你如何實作良好的檢測。

遙測的詞彙：日誌、指標與追蹤

在深入探討檢測策略之前，先了解你的應用程式可以發出哪些類型的遙測資料會很有幫助。目前先姑且參考即可（我們將在下一章詳細說明遙測資料型別），但大多數

剛接觸現代可觀測系統的讀者，會將這些資料稱為可觀測性有時所謂的「三大支柱」：日誌、指標與追蹤。

本書雖然不特別推崇「支柱」這個概念，但我們仍想強調這些資料類型對於可觀測性遙測的重要性。每種類型各有其特定用途，了解在檢測應用程式時該選用哪一種資料類型是很重要的。

從歷史上看，這些資料類型各自服務不同目的，回答有關系統行為的不同問題。在本章中，我們會先說明這些資料類型過去的用途與意涵，以及如何收集它們（之後再進一步深入拆解）。

日誌代表應用程式在特定時間點所發生的離散事件。日誌內容可能記錄用戶何時登入、付款何時處理完成，或錯誤何時發生。傳統日誌通常是為了讓人閱讀而設計的非結構化文字，但現代日誌實務則強調結構化日誌，採用一致的鍵值對，讓機器能輕鬆解析與分析。一筆結構良好的日誌條目可能長得像這樣：

{
“timestamp: “2024”,
“user_id”: “630630”,
“action: checkout_completed”,
[...]
}

指標代表一段時間內系統的數值測量結果。1

它們回答的問題像是「我的服務每秒處理多少請求？」或「我的 API 第 95 百分位的回應時間是多少？」指標通常以聚合或彙總資料的數值形式擷取。與其儲存每一筆個別請求的資訊，你可能會儲存一個隨每次請求遞增的計數器，或是一個追蹤回應時間分布的直方圖。這種聚合方式讓指標在儲存與查詢上更有效率，但也意味著你失去了深入探究該總和背後脈絡的能力。

分散式追蹤（也簡稱為「追蹤」）代表單一交易在系統中流動的完整歷程。當用戶在你的網站下單時，這個請求可能會觸發對數個相依服務的呼叫，用來更新庫存、計算運費、處理付款以及寄送收據電子郵件。追蹤會將所有這些操作擷取為一個相連的序列，向你呈現發生了什麼事，以及時間花在哪裡。這個序列中的每一個步驟稱為跨度，而跨度本身也可以包含關於該操作期間發生了什麼事的詳細資訊。

1 關於指標基礎資料類型的更多細節，請參見 OpenTelemetry Metrics Specification。

如同第 1 章所述，這些資料類型在歷史上一直是分開管理與儲存的，儘管它們其實互為補充。每種遙測類型都有其最適合的使用情境。日誌能提供你關於個別事件的詳細情境；指標能提供你系統健康狀況與效能趨勢的聚合視角；追蹤則能讓你端到端地檢視橫跨多個服務的複雜操作。

在下一章中，我們將探討如何優化並統一這些使用情境，以達成高度的可觀測性。目前，讓我們先專注於現代的檢測方法，這些方法能讓你輕鬆發出兼具多重用途的豐富遙測資料。

OpenTelemetry：可觀測性的通用語言

在過去，可觀測性廠商要求你使用他們自家的專有代理程式與函式庫，才能從應用程式中收集遙測資料。如果你想嘗試不同的廠商，或是將資料傳送到多個後端，就得重新思考整個檢測策略，而且通常還得面對必須拆除並替換現有代理程式與程式碼的窘境。這造成了令人沮喪的鎖定情境，你最初選擇的可觀測性工具，決定了你能如何檢測自己的程式碼。

OpenTelemetry（簡稱 OTel）應運而生，成為這個問題的解決方案。這是一個開源專案，提供廠商中立的 API、函式庫與工具，用於從應用程式中收集遙測資料。你可以把 OTel 想成一個通用翻譯器，讓你的應用程式說出任何可觀測性後端都能理解的共同語言。

OTel 的威力來自於它對關注點的分離。你的應用程式碼使用 OpenTelemetry API 來發出遙測資料，但這些資料實際傳送的目的地是可設定的。你可以同時將相同的檢測資料傳送到多個後端，這讓評估不同可觀測性工具或維持備援變得容易。你甚至可以完全更換後端，而不需要動到應用程式碼，只要更新目的地設定即可。因此，函式庫作者也能安心呼叫這個 API，因為他們知道除非經過設定，否則它預設不會執行任何操作，也就不會產生額外負擔。

OTel 開箱即用，就支援對熱門框架與函式庫進行自動檢測。也就是說，你只需極少的程式碼變更，就能讓基本的遙測資料開始流動。對於一般的網頁應用程式而言，你可能只需要加入幾行設定，就能自動擷取 HTTP 請求、資料庫查詢，以及對外部服務的呼叫。自動檢測讓你能立即檢視常見動作，以理解應用程式的行為，而無需進行大量的手動檢測。

但 OTel 真正的優勢，要等到你開始加入針對自身業務邏輯的自訂檢測時，才會顯現出來。自動檢測擷取的是應用程式技術層面的管線，而自訂檢測擷取的則是

與您業務相關的特定領域操作。舉例來說，自訂檢測可以用來捕捉用戶註冊流程、訂單處理流水線，或內容推薦演算法的表現。企業應用程式的簡單事實是，它們有許多獨特之處，自動檢測根本無法也不會理解這些細節。

許多工程師必然會問，究竟應該使用自動檢測還是自訂檢測。答案是兩者都要用。自動檢測與自訂檢測之間的區別，代表了捕捉遙測資料的不同方式，各有其優勢與適用情境。

自動檢測的運作方式是攔截應用程式框架與函式庫中的常見操作。當您發出HTTP 請求、查詢資料庫，或呼叫外部服務時，自動檢測可以在不需要您撰寫任何明確程式碼的情況下，捕捉這些操作的細節以及彼此之間的關聯。現代的自動檢測相當複雜，能夠理解複雜的框架，並以最少的設定擷取有意義的遙測資料。

自動檢測的主要優勢在於，只需極少的努力即可涵蓋廣泛範圍。您只需加入幾行設定，就能對整個應用程式中數十種常見操作進行檢測。這樣極少的努力就能讓您立即看見應用程式基本技術健康狀態的資料，例如回應時間、錯誤率、依賴關係與資源使用情況。

然而，自動檢測只能捕捉它所理解的技術操作，而且僅限於所有應用程式的最低共通標準。它無法理解您的業務邏輯、特定領域的工作流程，或應用程式特有的錯誤情況。舉例來說，自動檢測可能會告訴您某個資料庫查詢花了 200 毫秒，但它不會告訴您這個查詢是用戶註冊流程的一部分，而且該流程因為Email 地址已被使用而失敗。

自訂檢測填補了這個缺口，捕捉與您應用程式業務邏輯相關的遙測資料。這正是讓您能夠理解用戶行為、業務流程與特定領域操作狀態的方法。舉例來說，自訂檢測可以捕捉用戶何時完成特定工作流程、背景工作何時處理特定類型的資料，或業務規則何時觸發特定決策。

最有效的可觀測性策略會結合自動檢測與自訂檢測。自動檢測提供了全面的涵蓋範圍，

技術管線和整體如何協同運作的骨架。自訂檢測則將對業務運作和用戶體驗的洞察附加上去，為那副骨架增添肌肉。

團隊在生產環境中首次啟用自動檢測時，可能會大吃一驚——細節程度可能多到讓人難以招架，甚至適得其反。自訂檢測不只是要多加東西，也在於懂得取捨。培養對不同環境中什麼才有價值的直覺，能帶來豐厚的回報。舉例來說，記錄每一次資料庫呼叫和 Redis 查詢在本機環境或許有用，但在生產環境中，那些微小的跨度累積起來會對性能造成實質衝擊。

隨著系統日趨成熟，你可能會發現自訂檢測對於理解應用程式中對業務最重要的層面愈來愈有價值。我們會在第 7 章更詳細地探討這整個主題，並針對常見與較少見的應用程式架構提供具體的檢測指引。

檢測的所有權

在可觀測性計畫的過程中，團隊要做出的最重要決策之一，就是由誰擁有檢測策略，以及這套策略要如何隨時間演變。在傳統監控中，檢測往往是專責維運團隊的責任，他們會在應用程式部署後才加入監控。這種被動的做法常導致覆蓋範圍出現缺口，而且檢測與應用程式實際的結構及業務邏輯搭配得也不好。

現代可觀測性需要所有權上的轉變。撰寫應用程式程式碼的工程師，最能理解哪些檢測會最有價值。他們知道哪些操作對業務至關重要、哪些程式碼路徑最複雜或最容易出錯，以及在除錯時哪些上下文會特別有幫助。當開發人員擁有檢測時，它就會成為設計流程的一部分，而不是事後才補上的東西。

然而，這種所有權的轉變並不代表每位軟體工程師都得成為可觀測性工具的專家。相反地，成功的組織會建立起模式與慣例，讓開發人員無需深厚的專業知識也能輕鬆加入良好的檢測。這些慣例可能包括提供標準函式庫，以更容易地推廣常見的檢測模式；確保新功能包含適當遙測的程式碼審查準則；或是說明何時、如何加入自訂檢測的文件（更深入的探討，請參見第六部分關於可觀測性治理的內容）。

你的檢測策略應該隨著應用程式一起演進。隨著系統日益複雜、隨著你從生產環境事故中學到經驗，以及隨著你的

隨著你對用戶行為的理解逐漸加深，你的檢測需求也會跟著演變。換句話說，檢測並非一次性的設置工作。身為軟體工程師，你應該將檢測的演進納入開發流程中，確保你的可觀測性能力能夠跟上系統複雜度的腳步。

隨著時間推移，你會將檢測建立為開發流程中的主要交付項目，就像測試一樣（參見第 9 章，其中討論了測試驅動開發與可觀測性驅動開發之間的異同）。在規劃新功能時，你的團隊應該明確討論哪些遙測資料能幫助你們最了解這些功能在正式環境中的行為表現。在進行程式碼審查時，你的團隊應該評估提出的變更是否包含足夠程度的檢測。在進行事後事件回顧時，你的團隊應該找出遙測資料的缺口，這些缺口曾使調查工作變得比原本更加困難。

如果尚未如此，檢測將成為你持續進行的軟體開發工作流程中不可或缺的一部分。

建立自訂檢測策略

要制定有效的自訂檢測策略，需要有系統地思考應用程式行為的哪些面向最值得了解。與其對所有東西進行檢測，不如把焦點放在能見度對你操作和改善系統的能力影響最大的領域。

首先，找出應用程式中的關鍵用戶旅程（參見第 11 章）。關鍵用戶旅程是指用戶必須成功完成、才能讓你的業務運作的一連串操作。當然，具體細節會因產業和使用情境而異。但舉例來說，假設你有一個電子商務應用程式，對你來說，業務關鍵的用戶旅程可能包括用戶註冊、產品搜尋、購物車管理和結帳等項目。或者，假設你有一個內容管理平台，那麼關鍵用戶旅程可能包括內容建立、發布和消費等工作流程。對這些旅程進行檢測，能讓你看清核心業務運作的健康狀況。

考量你所屬領域特有的錯誤狀況。雖然自動檢測可能可以捕捉到 HTTP 500 錯誤，但它無法理解像是「庫存不足」、「無效的優惠券代碼」，或「用戶不在此操作對應的正確實驗組別中」這類特定於業務的失敗情況。自訂檢測可以捕捉這些特定於業務的錯誤狀況，並附上識別、理解及解決這些問題所需的上下文。務必區分暫時性與永久性的失敗，以及哪些失敗是 IT 團隊真正可以採取行動處理的、哪些不是。

當你剛開始思考該捕捉什麼時，可能需要一些練習才能上手。剛開始時，不妨想想在事件回應或分析系統效能時，你經常需要回答哪些問題：

• 哪些用戶受到影響？•

• 我的流水線有哪些關鍵階段？•

• 目前正在使用哪些 feature flag？•

請記住，檢測不只是要捕捉發生了什麼事。檢測的存在，是為了捕捉能幫助你理解「為什麼」發生這件事的額外背景脈絡。應納入相關的業務背景資訊，例如用戶區隔、當時啟用的 feature flag，或請求的來源（更深入的範例請見第6 章）。這類背景脈絡資訊，往往正是快速解決問題與耗費數小時嘗試重現並理解問題之間的差別。

隨著你的自訂檢測不斷演進，你會想建立一些模式與慣例，讓所有開發者都能輕鬆加入一致的自訂檢測。這可能包括所有業務操作追蹤都應納入的標準屬性、命名慣例，或是針對常見模式（例如背景工作處理或 API 端點）的檢測範本（更多提示請見第 31 章）。

請記住，隨著你對這些概念愈來愈深入，以及對系統理解愈來愈深，你的自訂檢測需求也會隨之演進。在發生重大事件之後，你會開始發現一些新的檢測機會，這些機會若事先具備，本可幫助你更快偵測或解決類似的問題。當你發現自己一再翻找日誌，或手動關聯資料以理解系統行為時，你很可能就會找出能簡化這類分析的新自訂檢測。

目標是建立不僅能幫助你理解系統中正在發生什麼事的檢測，還能讓你有能力提出並回答那些你甚至還沒想過的更好問題。回想一下，具備高度可觀測性的系統，能讓你理解並解釋系統可能進入的任何狀態，無論其多麼新奇或複雜。優秀的檢測能捕捉到高層次的背景脈絡，並為卓越的可觀測性奠定基礎。

話雖如此，在建立自訂檢測時，很容易會想要捕捉一切可能的資訊。雖然有些可觀測性工具是為配合這種策略而設計，但對其他工具來說，這種做法可能並不實際。時間序列資料庫有明確的設計限制，當某個指標的基數過高時，會導致不佳的結果。全文檢索資料庫則需要建立索引與預先運算好的結構描述（schema）。過多的檢測會產生雜訊，反而讓

找出你真正需要的訊號，同時可能增加成本與營運負擔，卻無法帶來對等的價值。

成本考量、資料量管理與處理

無可避免地，你最終需要管理適當檢測後的應用程式所產生的大量遙測資料。事實是，大部分流量都平淡無奇、司空見慣，這類流量對應的資料也是如此。但有些流量卻很有意思，或者對理解正在發生的事至關重要。要決定如何依此原則管理資料，並非簡單的計算問題，而你也有許多可運用的手段來進行這類管理，並在各種取捨之間求取平衡。我們不會在此深入涵蓋所有選項，但會介紹基本概念與考量重點。這些主題會在後續章節中進一步討論。

處理與流水線

無論你使用哪種檢測方式，在儲存遙測資料之前先進行處理是很常見的做法。舉例來說，你可能需要捨棄或轉換資料以縮減其資料量，或是將資料萃取、轉換為其他格式。你也可能需要做一些事，例如在把遙測資料傳送到後端儲存之前，進行資料正規化或過濾掉個人識別資訊（參見第 16 章）。

在程式碼中撰寫檢測時，為了可攜性考量，你應該對後端能力保持中立，並盡量把處理決策延後到遙測收集過程的較後階段再做。畢竟，你並不確切知道在調查過程中會需要哪些遙測資料。如果所擷取的某些資料日後證實有問題，移除發現用不到的特定細節，會比一開始就沒有收集、事後想要卻求之不得的屬性容易得多。

此外，你也應該考量檢測策略、設計與實作在擴展性方面的影響。舉例來說，並非所有使用某個檢測程式庫的用戶都需要全域複寫，或是次毫秒等級回應時間的遠端程序呼叫。有些用戶需要，但並非全部都需要。良好的檢測設計會將這種差異性納入考量。在這類情況下，你可以使用像是遙測分層（telemetry layering，參見第 7 章）這樣的技術。

成本考量與資料量

你可以擷取任何你想要的遙測資料，但這要花多少成本？你的檢測策略中幾乎所有事情，都可以用成本來量化。

有些成本很明確，例如儲存、記憶體或查詢資料所需的運算量。如果你購買了可觀測性解決方案，這些成本通常會以外部供應商帳單的形式呈現。或者，如果是自行建置的可觀測性解決方案，這些成本則會以建立與維護所需的基礎架構與人力等內部成本來體現（見第 29 章）。

許多成本則較為隱藏。舉例來說，不良的檢測策略可能導致資料混亂或資料集分散不一致。如果資料散落在 10 到 15 種不同的工具中，你很可能會付出隱藏成本，例如警報觸發後，事故應變人員無法快速分類並正確診斷問題所造成的認知負荷與摩擦。這些成本通常會以事故解決時間延遲的形式明顯呈現出來，但造成延遲的背後因素往往對高層利益相關者而言是不可見的。此外，成本管理實務本身也會對開發者生產力與事故解決帶來額外負擔——例如試圖釐清為何跨度被取樣排除，或建立繁瑣的解決辦法來降低指標基數。

在我們探討購買與自建可觀測性解決方案之間的抉擇時，會再進一步討論這些成本。目前，請先記住天下沒有白吃的午餐這個道理。決定自建或購買的團隊，可能會把人力和基礎架構等明顯成本列入考量，卻經常對不那麼顯而易見的成本感到意外。舉例來說，自架的遙測資料儲存通常仍會產生像網路出口流量這類成本，而這類成本在可用區域、地區、叢集、資料中心等之間傳輸遙測資料時，可能會迅速攀升。

在大多數情況下，收集並透過網路將少量遙測資料傳送到後端系統所需的額外負擔相當小且易於管理。然而，在大規模運作時，分散式或雲端原生的工作負載可能會把這股涓涓細流變成資料洪流，尤其是在檢測設定錯誤的情況下。目前已有多種解決方案可用於大規模管理遙測資料，例如利用遙測流水線在本地緩衝高解析度資料，同時只透過網路傳送彙總後的資料（相關概念請見先前的說明框）。

不過，在這個入門章節中，有一項成本控制與流量管理的特定要素值得特別關注：遙測取樣。

取樣的基本概念

取樣並非記錄所有產生的遙測資料的 100%，而是選擇性地決定要保留哪些資料（即所收集的樣本），並捨棄其餘的資料。整體而言，我們認為無論在何種規模下運作，對所有遙測資料進行取樣都是一項最佳實務：取樣讓你能夠擷取真正有價值的資料，同時捨棄無關的雜訊。

從概念上來說，資料取樣只有兩種方式：可以在資料發出之前取樣，也可以在資料發出之後取樣。

之前

在資料被發送出去之前進行取樣有多種方式。舉例來說，有一種很常見的預發送遙測取樣方法是大多數開發人員經常沒有意識到的：指標。指標其實是一種取樣的形式。它們是透過選擇量測值與維度，然後將特定時間區間內發生的所有量測值加總計算而得的數值。在該區間內發生的個別事件被視為雜訊而被捨棄，只留下預先聚合後的摘要值。

更常見實現的形式是類似基於機率的頭部取樣（head sampling），這是一種簡單的技術，簡而言之就是「只取樣我的追蹤或日誌的一定百分比」。頭部取樣對於真正處於高流量的服務效果很好，但這種簡單的基於機率的方法對於非超大規模運作的服務會帶來明顯的缺點。

假設你的服務具有統一的流量模式，而你只關心聚合統計資料。如果是這種情況，這種方法會傾向於降低整體遙測資料量，但代價是可能會漏掉異常值或長尾問題。換句話說，在我們只保留 1% 遙測資料取樣的情境下，一個只有0.1% 發生機率的錯誤完全有可能被徹底漏掉。同樣地，對於超大規模運作的服務而言，0.1% 的流量仍然是龐大的流量，但對於規模較小的服務，你就有可能因為頭部取樣而遺失有價值的錯誤資料。

在資料發送出去之前（也就是在來源端）進行取樣，通常傾向於以資源效率換取資料的忠實度與完整性。它也完全不會顧及其決策上游或下游的相關情況。舉例來說，在做出頭部取樣決策的當下，根本無從得知下游服務呼叫是否會導致錯誤。如果你追求的是高度的可觀測性，頭部取樣的這種取捨幾乎完全偏向於節省成本，而犧牲了可除錯性。

之後

第二種取樣形式發生在遙測資料被發送出去之後。與頭部取樣不同，這種取樣形式應該用來優化遙測資料的忠實度與完整性。有多種技術可以用於此目的。

舉例來說，在分散式追蹤中，你可以使用尾部取樣（tail‑based sampling）。尾部取樣是一種技術，在做出取樣決策之前，會先收集並緩衝一條追蹤中的所有跨度。如此一來，你就可以套用處理邏輯與規則，檢視整個交易記錄的全貌，以判斷這筆特定記錄是否值得保留。

換句話說，透過尾部取樣，你可以在做出保留或丟棄的決定之前，完整檢視每一條追蹤或日誌行。這條追蹤看起來是否就像過去一小時內記錄的數千個其他200 OK 狀態請求一樣？還是它捕捉到了某些更重大、值得保留的內容？

有效遙測取樣的技術相當複雜且細膩，因為它們最終必須考量到你業務的特定規則或考量因素。就本入門內容而言，我們不會進一步詳細說明如何有效取樣。但我們認為必須指出，並非你產生的所有遙測資料都值得保留。請記住，所有遙測資料都有一定程度的成本。若運用得當，取樣是你可以用來管理可觀測性成本最有效的槓桿之一（深入技術請參閱第十五章）。

結論

你無法改善你無法衡量的東西。檢測是你可觀測性體驗的基礎基石。每位軟體工程師都必須學會將檢測整合到開發生命週期的每個環節。你將依賴自動檢測來快速呈現應用程式運作健康狀態的基本徵象。你也會依賴自訂檢測來瞭解業務功能的健康狀態。

你需要從整體角度思考檢測，並針對其所有權與演進形成一套連貫的策略。你很可能會透過 OpenTelemetry 將檢測加入到你的程式碼中。身為軟體工程師，你或許不會參與管理可觀測性成本的所有細節，但你至少應該對所使用的技術及其影響有一些基本瞭解。本章介紹所有工程師都應該知道的概念，我們會在本書其餘部分持續進一步說明。

接下來，我們將探討另一個與遙測相關的概念，這對於學習具有高度可觀測性的系統來說是至關重要的基礎。唯有消除你所知的日誌、指標與追蹤這幾種遙測類型之間的錯誤區分，才有可能從三大支柱模型邁向統一的可觀測性模型。
