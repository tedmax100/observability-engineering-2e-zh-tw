第 18 章
CI/CD 流水線的可觀測性

本章由 Namespace Labs 執行長 Hugo Santos

與 Honeycomb 技術院士 Liz Fong‑Jones 共同撰寫

在本書前面，我們已經說明了可觀測性如何加速回饋循環，讓團隊能夠了解需要對軟體做出哪些改動。但團隊在完成這個回饋循環時，往往會被一個障礙卡住：實際取得開發中程式碼的回饋、或是把程式碼推上生產環境所需的時間。如果你的感覺就像是在手動操作火星探測車，輸入指令與看到結果之間有長達數分鐘的延遲，那麼無論你能多快分析並得出結果都無濟於事。可觀測性與可控性是一對相輔相成的概念，你必須確保這兩個回饋循環都取得平衡。

為了明確定義本章將會使用的詞彙，我們將 CI 定義為從原始碼與輸入內容建立建置產出物（build artifact）的流程，而 CD 則是將該建置產出物部署到生產環境的流程。這兩者都是完成「撰寫程式碼」到「在生產環境中執行」這段回饋循環不可或缺的環節，但不同組織在這兩個流程上可能各自具備不同程度的複雜性。

在本章中，你將學習以嶄新的角度審視 CI/CD 的基礎，並了解投資可觀測性如何能協助你提升 CI/CD 流程的性能。我們將以一個案例研究作結，說明Honeycomb 如何運用可觀測性進行資料蒐集，藉此找出正確的創新切入點以提升建置性能。稍後在第 23 章中，你將進一步了解「回饋循環的回饋循環」，以及收緊每一個回饋循環如何能確保你的團隊快速且可預期地交付軟體。

為何可靠且快速的 CI/CD 至關重要

假設你的團隊中有一位開發者剛完成一項功能的撰寫，想要將它落地並部署到生產環境。第一步是提交一個 pull request。你知道他們接下來要做的事情就是等待測試通過。而在測試通過之前，測試與生產環境的二進位建置都必須先完成。由於那則關於「我的程式碼正在編譯！」的 xkcd 漫畫在 2026 年依然貼切，他們會利用這段時間去喝杯咖啡等待。現在，手裡拿著合適的飲料，他們卻看到有一個測試失敗——結果重新執行後又通過了？這大概是個不穩定的測試（flake），可以放心忽略（至少團隊裡的傳說是這麼說的）。

你把建置做得越快、越具確定性，就越能快速地將變更推送到生產環境，透過可觀測性工具取得真實的反饋。接著，透過進行必要的變更、推送出去，並建立起穩定的節奏，良性循環便會發揮作用。

CI/CD 流水線的核心，其實就是為開發者提供反饋的生產系統。開發者的時間很昂貴，因此把 CI/CD 視為次要事項，或是要求它低於生產等級的嚴謹程度，都會導致週期的浪費。然而，它們卻常常被當成無法管理的不透明黑箱：只要建置成功，系統就「沒問題」。但如果你的 CI/CD 平台對你這個內部用戶來說是不透明的，你將永遠不會知道為什麼建置很慢、是什麼原因造成了不穩定的測試（flake），或該在哪裡介入，更遑論它到底花了多少成本。

在所有可觀測性應用中，建置可觀測性擁有最佳的投資報酬率

你的平台工程團隊可以透過識別並消除快速將軟體交付到生產環境的瓶頸，來放大自身的影響力。雖然人為的回應時間對程式碼審查以及品質保證（QA）測試所涉及的任何人工流程來說很重要，但一旦 QA 自動化之後，開發者撰寫程式碼到該程式碼真正服務生產環境用戶之間的延遲，大部分時間都花在編譯、測試以及經歷各個部署階段上。

你的公司可能營運著每秒處理數百萬個請求的服務，其中每個請求的價值可能只有幾分之一分錢。若不實作取樣及其他減少遙測量的技術，很難看出加入完整檢測所帶來的投資報酬率。但建置系統與面向客戶的高 RPS（每秒請求數）系統完全不同——每個請求依定義都有一位開發者在等待，以便解除阻塞，因此請求失敗或緩慢，可能會浪費數十美元的運算資源，以及數百美元的開發者時間。

截至本文撰寫之時，開發程式碼以建立 pull request 的成本正在下降，但仍遠非零。LLM 與代理程式建立 pull request 需要耗費 token，因此也需要花錢。而開發者的成本更高，體現在他們的薪資上。

因此，觸發建置的請求數量會相對有限且數量不多，即使有 AI 輔助，每位開發者一天大概最多也只有十來個拉取請求（pull request）。若超過這個規模，就會犧牲人類對於所寫程式碼的理解，程式碼審查與品質檢查也會流於形式。

這種流量規模非常適合作為導入可觀測性的起點，讓你能親自體驗其效果，而不必等到在正式上線的生產服務中才實作。這樣的規模很容易自行架設並管理開源可觀測性工具，也很適合套用商業可觀測性解決方案的免費方案。而對開發者生產力帶來的回報可能非常可觀。

可觀測性能為這個盲點帶來曙光。它讓團隊與組織能夠根據真實資料來推理性能、可靠性與效率，而不只是憑直覺或組織內部的既有認知（例如「忽略那個測試沒關係」）。

從可觀測性的角度看 CI/CD

要回答問題，首先需要資料。我們可以從簡單的地方開始，看看有哪些動詞與名詞，以及該如何衡量它們。這既能複習 CI/CD 系統的基本元素，也能作為將可觀測性套用到這些基本元件的指南。

持續整合與持續部署的本體論

在第 17 章中，你學到了本體論的重要性，以及定義清楚的動詞、名詞與形容詞來描述系統，並為事物指定一致名稱的重要性。現在，讓我們把這套方法套用到 CI/CD 系統，看看它與可觀測性的本體論如何重疊。

每個系統都有各自的協調（orchestration）構件與工作單位。GitHub Actions 有 workflows、jobs 與 steps。Buildkite 有 pipelines、builds、jobs 與 steps。而開源系統 Argo 與 Concourse 則有 applications、projects與 states 等單位。Workflows 會連結回拉取請求，並提供開發者等待的關鍵訊號。表面上看似「我的測試是否通過？」的問題，實際上是「workflow X、Y、Z 是否通過？」。每個 workflow 接著再細分為多個可並行執行的 jobs，只要各自的先決條件都被滿足即可。

在 CI 的世界中，每個 Job 可能有多個在該 job 內依序執行的 steps。而每個step 實際上可能是執行測試框架中包含多個測試的部分測試套件的一道指令。至於 CD 系統，jobs 則可能包含取得憑證、推送設定檔、變更生產環境等steps，或以其他方式促使變更部署到生產環境，使期望狀態與目前的生產狀態一致。

幸運的是，這些概念也能很好地對應到生產環境可觀測性世界中的跨度與追蹤。每個工作流程就是一條追蹤。每個作業是一組隸屬於該工作流程根跨度之下的跨度。而作業內的每個步驟則是該作業內含的一個跨度。此外，你很可能也擁有執行每個步驟所產生的詳細日誌。

如果你是開發人員，你能控制的選擇往往落在執行層面，而非編排層面。這通常意味著執行你的測試程式碼與建置二進位檔的作業。而且因為它們只執行你的程式碼，所以最容易讓你優先改善並提升自己的體驗。

身為平台團隊，你可能擁有額外的施力點——例如設計編排方式，以及編排要如何、在何處運行、針對哪些工作單元。舉例來說，你或許能設計批次處理系統，將多個拉取請求（pull request）一起執行，只有在其中一個請求失敗時才將它們拆開。或者，你也可以依據是否修改了相關程式碼來有條件地執行測試。不過，我們建議你先從簡單做起，只有在能帶來顯著生產力提升時才增加複雜度。

檢測基礎

一旦你為 CI/CD 系統建立了清楚的本體論，自然的下一步就是開始在你的可觀測性工具中蒐集資料。首先，從你的 CI/CD 解決方案匯出個別作業執行的資料：它們何時開始、何時結束，以及是否成功。幸運的是，這與更廣泛可觀測性世界中的追蹤跨度有著直接的類比關係。每個作業都有名稱、有開始與結束時間（或開始時間與持續時間）、隸屬於某個工作流程中的一組作業集合，並且可能還有其他值得追蹤的屬性。

最重要的遙測資料之一，就是正在通過 CI 的是哪個儲存庫（repository）與程式碼分支（branch）。分支之所以重要，是因為它們在開發人員將變更帶入程式碼庫的方式中扮演著關鍵角色。舉例來說，測試是針對 main 分支還是拉取請求分支執行，可能會影響該組特定工作流程呼叫的重要程度，以及你的性能標準應該是什麼。如果有人寫了一個不當地慢或不穩定（flaky）的測試，那只會拖慢他們自己的呼叫，而不會影響團隊其他人。然而，如果那項變更被合併了，導致現在每個人的建置都變慢，包括 main 分支的建置，那你現在就有個嚴重的問題了。

在這裡，你可以依賴你的 CI 系統所提供的整合，或者你也可以自行建立整合。通常會有明確定義的「第一步」與「最後一步」由你所控制，可以在其中發送事件（例如「這個作業已開始」與「這個作業已結束」）。

如果你使用像 GitHub Actions 這樣的平台，匯出遙測資料可以簡單到只需使用一個能將 Job 與 Step 持續時間匯出到你可觀測性系統的 OTel action

堆疊。有了這些基本指標所提供的資訊，你的團隊便能開始推導出預期行為，也應該能夠回答諸如「這應該花多久時間？」以及「這項工作是否經常失敗？」之類的問題。

定義服務水準指標並實現可預測性

典型成功率與整體延遲的資料，可作為你最初隱含的 SLI 與 SLO。這樣的理解能將 CI/CD 系統從一個謎團轉變為一個可測量的流程，讓你能夠追蹤其行為並針對變化採取行動。與其被動接受這個不透明黑箱給出的結果，你可以理解它並掌控它。至少，設定這些預期值能讓你偵測並預防退化，即使你尚未準備好投入資源全面翻修整個流水線。

有了資料的加持，對話的方式也隨之改變：與你的用戶（開發團隊）溝通時，回饋不再只是基於「感覺」，而是基於真實資料（例如「工作耗時一直在惡化」）。你也能向領導團隊展示真正的問題所在，並激發修復所需的工作動力。

在 CI 系統中，我們傾向於將性能定義為「從開始到變更被驗證完成」所需的「端到端用戶時間」，而這最能貼切反映用戶的體驗。開發人員在等待軟體驗證完成，以確認他們的變更已經完成，程式碼可以提交到儲存庫中。就像在正式環境系統中一樣，你需要仔細思考並定義對你的用戶而言什麼才是重要的（以及你的用戶究竟是誰）。

即使某些指標不會直接影響 CI/CD 執行的典型延遲，你可能仍會想要追蹤那些影響成本或可預測性的指標。舉例來說，你可能希望確保只有一小部分的測試及其相關執行是不穩定的（flaky，以成功重試的百分比衡量），或是透過追蹤主分支上的測試失敗率，確保你的 CI 提交前檢查（presubmit hooks）運作正常。至於成本方面，資源使用量能幫助你確保消耗速度符合 CI 供應商合約承諾的可接受範圍，以避免因浪費合約支出或意外收費而產生意外狀況。

從工作到有向無環圖與追蹤

在工作（job）層級收集資料是最簡單的切入點，因為我們掌控著作為工作一部分執行的程式碼。但我們先前已經知道，工作只能呈現出故事的一部分。提交請求（pull request）等待的是整個工作流程完成，而非構成該流程的個別步驟或工作。雖然工作內的步驟是循序執行的，但 CI 平台中的工作流程最終會演變成由多個工作組成的有向無環圖（DAG），其中每個工作只能依賴於在它之前的那些工作。

這些 DAG 往往是自然而然形成的。因為每次想要產生一組不同的測試時，都重新建置用於測試的容器映像，成本相當高昂，所以你

新增一個預建（prebuild）任務，只建置一次該容器映像檔，並讓後續任務都能使用它。你現在有了一個相依關係：如果預建任務變慢，所有依賴它的任務也都得等待。如果預建任務失敗，後續任務就無法執行。而後續任務若無法執行，你就沒有它們的資料可供分析。

透過將 CI/CD 本體論對應到分散式追蹤的基本元素，你可以跳脫團隊對建置緩慢的口耳相傳，轉向一個可量化的流程，讓每個工作流程都有可驗證的結果。然而，孤立地衡量單一任務只能說明部分故事。開發人員等待的不是個別步驟，而是整個驗證序列的完成。在現代交付流程中，這些序列自然會演化成一個由相互依賴的任務組成的有向無環圖（DAG）。要理解建置為何真的變慢，我們必須跳脫任務清單，將這個圖以追蹤的形式視覺化，藉此找出關鍵路徑（critical path）——唯有在這裡進行最佳化，才能真正縮短回饋迴圈。

進行改善並加以衡量

隨著你的產品基礎架構演進，這些工作流程往往會變得更複雜。你不會只有一個容器映像檔的預建，而是有五個。你會有具備多個相依關係的任務，以及依賴這些任務的其他任務。此時你執行的是一系列具有明確定義相依關係的任務，而成功的定義是所有終端任務都成功完成。拆分成不同的任務讓你有機會針對高成本的步驟採用流水線與平行處理。這樣的複雜度為何值得？又該如何衡量結果？

流水線化是提升 CI/CD 平台性能的重要特性，但也伴隨著取捨。在完全序列化的世界裡，每個步驟依序執行，很容易理解工作流程進行到哪個階段。但建置可能耗時數十分鐘，而端到端的測試套件可能耗時數小時。如果將這些步驟序列化，可能要很久才能開始部署。

因此你依靠流水線化，最終讓多個高成本任務平行執行。此時，孤立地觀察單一任務的性能與行為，已無法說明全貌。舉例來說，如果你的系統由 React 前端、Go 後端二進位檔和 Rust 後端二進位檔組成，負責 Go 後端二進位檔的團隊可能會反映他們的建置變慢了。也許他們開始依賴一個更大的 Go 函式庫，因而在編譯階段花費更多時間。但若拉遠來看，如果 Rust 建置主導了整體執行時間，那麼 Go 建置上的細微差異最終並不會影響整體工作流程的性能，總耗時仍維持不變。

所以沒錯，你的 Go 建置雖然很慢，但真正的問題出在別處。你剛剛找到了自己的關鍵路徑。就算你花時間優化 Go 建置，端對端的時間也不會因此改變。

了解性能：把持續整合當成生產環境對待

CI 工作本質上就是有截止期限的生產工作負載。雖然擷取各步驟的耗時有助於我們看出建置正在變慢，但要縮短關鍵路徑，需要用我們在生產環境分析性能時同樣嚴謹的鑑識手法。就像任何生產系統一樣，你的流水線可能受限於 CPU、記憶體、每秒輸入／輸出操作數（IOPS），甚至只是單純的輸入／輸出（I/O）（例如呼叫其他系統）。你的生產工作很可能運行著與測試工作相同的程式碼庫，但額外承受著要在短時間內完成大量工作的壓力。

要擺脫「用猜的」來判斷工作為何緩慢，我們必須能夠將追蹤與主機層級的指標、日誌相互關聯，藉此找出工具鏈碰觸到極限的地方。唯有透過實驗與蒐集資料，才能得知真相。

以下是實驗如何揭露瓶頸的一些例子：

• 有一個團隊假設增加運行器資源（也就是核心數）就能解決 JavaScript 建置緩慢的問 • 題。他們的檢測資料顯示，建置時間並沒有變化。真相是這項工作使用了 webpack，卻沒有必要的設定（例如 ThreadLoaders）來受益於平行處理。多出來的核心閒置著，因為建置仍然是單執行緒運行。

• 有一個團隊假設，在目前的硬體上，他們的 Go 建置已經達到最快速度。在為 Go • 工作提供更多核心進行試跑後，他們發現建置時間縮短了一半。這揭露了 Go 對平行處理的原生支援讓它具有高度彈性，用更多 CPU 換取速度是可行的做法。

• 有一個團隊儘管擁有非常高效的增量建置，卻始終無法快速地將發布推送到生產環 • 境。他們是從單一來源工作把容器映像推送到所有地區，結果受限於網路頻寬，而非主機資源。

藉由有效運用對建置系統的可觀測性，你就能優化真正重要的事，而不是猜測接下來該改進什麼，或是承認失敗。

可預測性：真實世界的權衡

我們所依賴的許多與建置和測試相關的系統，都是圍繞著「從一組定義明確的輸入產生可預測輸出」這個概念所建構的。當這種關係能夠被信任時，就可以被快取。

舉例來說，拉取同一個容器映像（其摘要值已知）將永遠得到相同的位元組。這項特性並非實作細節，而是一種正確性契約。也正因為這項契約，只需拉取一次相同摘要值的映像，並在多次執行中重複使用，並不會損及正確性。這是提升增量性的一種簡單又有效的方式。

這種模式在 CI 系統中隨處可見。如果輸入是穩定的，輸出是確定性的，你就可以安全地記住結果。而記住結果往往就是快速系統與⋯⋯嗯，緩慢系統之間的差別。

然而，可重現性並非憑空存在。外部限制與共享資源都可能限制它。在「以並行方式完成更多工作」與「為了成本效益而充分利用資源」之間，始終存在著一種張力。塞入更多工作往往會導致更高的資源爭用，而更高的資源爭用往往會導致更多不穩定（flaky）的測試或間歇性失敗的工作。到了那個地步，問題就不再純粹是技術問題了，而是變成「你的工程文化能否容忍不穩定測試？」

當你的測試具有可重現性時，因資源限制而產生的不穩定測試，實際上就等同於一種尾端延遲稅——重試並不會讓問題消失，只會增加耗時。要推理這些權衡，一個有用的方法是比較 P90 與 P50 延遲。單獨檢視個別工作或工作流程或許無法呈現這項資訊，但若將每次工作流程執行視為單一追蹤，並套用現代可觀測性技術，就能幫助你辨識 P50 與 P90 的趨勢，為你的開發者做出正確的選擇。

資源限制的管理在不同工具鏈之間也並不一致。有些生態系比其他生態系更容易處理這個問題。舉例來說，go test 讓橫跨額外核心進行擴展相對容易。而端對端瀏覽器測試則難以有效並行化得多，最終往往受限於記憶體、I/O 或外部服務，而非純粹的 CPU。

團隊用來緩解此問題的一種方式，是依賴短暫存在的專用資源。與其把不相關的工作塞在一起碰運氣，不如佈建具有已知容量的短生命週期環境。乍看之下這可能顯得成本較高，但一旦把重試、調查時間與開發者等待時間都算進去，往往反而更划算。

由於建置與測試往往具有高扇出（fan‑out）與批次導向的形態，你也應該預期，隨著扇出增加，可計數的失敗次數也會隨之增加。這種情況可能發生在你的工程團隊成長之時，也可能發生在編碼代理（coding agents）開始以更高速率產生變更之時。

接下來的關鍵問題是：這些是真正的失敗，還是基礎設施造成的失敗？建置與測試執行的可預測性越高，就越容易回答這個問題。舉例來說，隱藏在測試中的細微同步或計時問題，可能一直不會顯現，直到底層硬體平台改變、一切突然執行得更快時才浮現。

由於 CPU 是一種彈性、分時共享的資源，你的測試對計時變異的容忍度越高，就越能有效利用你的基礎設施。這種穩健性能直接帶來更高的打包率：用更少的 CPU 完成更多工作——而不必付出不穩定測試的代價。

為了推敲這些限制，請務必檢測作業系統層級的指標，或是與能提供這類資訊的供應商合作。你的作業（jobs）除了受 CPU 與記憶體限制外，還會受到許多其他限制：開啟的檔案描述符、核心共享記憶體、通訊端緩衝區大小等等。

在執行整合測試或端對端測試時，一個有效理解所有服務限制的方法，是依賴容器並收集 cgroup 層級的指標。這樣一來，你不僅能看到自己行程的資源使用情況，也能看到相依項目的整體資源使用量。

在這個層級，CI 可觀測性開始變得非常類似生產環境的可觀測性——這並非巧合。

（缺乏）漸進性：持續整合的存在之惡

如果你訪談一群開發者，可能會聽到不少對 CI 效能的抱怨。你甚至可能會聽到這樣的話：「我在自己的 MacBook 上 5 分鐘就能跑完整個測試套件，CI 卻要花 20 分鐘；我們能不能直接用 Apple 機器來跑 CI？」你當然可以，但那是另一個故事了。

CI 應該要快。為什麼它不快？

許多工作流程一開始很簡單，而且為了可重現性，每次都會從頭做同樣的工作。這其實是件好事：你想要可重現性。你想確保你的建置與測試每次執行結果都一致，不依賴其他作業留下的狀態。但每次都從零開始，代價非常高昂。

值得慶幸的是，許多工具鏈都有解決這個問題的方法：它們會保留中間檔案。使用語言與工具鏈特有的語意，可以保證一定程度的正確性，也就是只有由相同輸入產生的中間檔案才會被重複使用（前提是你有在工作流程執行之間保留這個快取）。

舉例來說，Go 就有模組快取與建置快取。如果你對 Go 程式的其中一個原始檔做了小幅修改後重新建置，建置速度會很快，因為其餘的輸出檔案都會被重複使用。善用這種漸進性

工具鏈（及其正確性保證）能讓你同時擁有兩全其美的成果：可管理的隔離性，加上出色的性能。

維持緊湊的建置性能、讓開發者保持愉快

當你能夠測量建置性能時，可能會開始發現尾部延遲（即你的 P90 工作耗時或更高）與不穩定性之間的關聯。對某些類別的測試而言，要找到完全的可重現性是困難的，而確保所有測試都有一致的行為，最終代價也會很高。

測試越複雜，牽動的元件就越多，出現不確定性的機率也越高。許多工作流程解決方案都有一個神奇的解法：重試。假設你有一個工作有 5% 的機率會失敗（你已經測量過），於是你啟用重試（可能是針對整個工作，有時則是針對工作中個別失敗的測試），失敗的情況幾乎就看不到了。但你的尾部延遲卻因此飆升。你的 P50 幾乎沒變，但有時候你最終要多等一倍的時間。只有把可觀測性整合進更廣泛的執行範圍中，你才能理解事情失敗的頻率，以及重試一個測試或步驟所帶來的延遲是否值得。

而這正是艱難的權衡所在。不同團隊會處於不同的成熟度層級，對於這是否是個問題也會有不同的判斷。對某些團隊來說，「大多數時候」都通過就已經足夠讓人有信心了。也許你還有其他方式可以測試所做變更的行為，能為你帶來額外的信心。

這裡有一個有趣的觀點：你的測試是生產環境的第一層代表。如果某個東西在你的測試中失敗，這可能是測試工具本身的因素，但也可能是一個真正的問題，只是要等到之後才會在生產環境中發現。

案例研究：快速建置時間在Honeycomb 的重要性

在 Honeycomb，我們相信要為自家的工程團隊確保出色的開發者體驗；這是我們唯一能與工程師人數遠多於我們的公司競爭的方式。我們一直以來都努力將建置時間維持在 15 分鐘以內，一旦建置時間超過這個門檻，就會採取強力措施。事實上，我們對這項資料設有內部的服務水準目標（SLO），只要超過5% 的建置耗時超過 15 分鐘，就會發出警報！

但事實證明，15 分鐘仍然不算是一個很好的體驗；開發者寧可切換任務，也不願等待建置完成，因而失去寶貴的思路脈絡。理想情況下，建置時間若能維持在六到七分鐘以內，就能讓開發者留下來看著建置完成，隨時準備快速修正CI 失敗的問題。這種追蹤實作細節的心智狀態──存在開發者腦中的那部分──就永遠不會被換出，因為他們不會覺得有必要在等待時轉去看 Slack 或去拿下一張票（ticket）來處理。

可觀測性是否也能在這裡幫上忙呢？

Honeycomb 改善建置時間的歷史

以可觀測性驅動來優化建置時間與可靠性，讓 Honeycomb 得以保持開發人員的愉快與生產力。可觀測性驅動開發不僅適用於程式碼，也適用於任何足夠複雜的系統！我們甚至認為，我們的建置系統和儲存引擎一樣，都是我們任務關鍵的複雜系統。如果建置系統當機，我們就無法部署新程式碼，開發人員的pull request 就會像牛奶一樣開始變質。未部署的程式碼是會腐壞的。

在 2018 年 6 月到 2019 年 5 月之間，建置 Honeycomb 所需的中位數時間從7 分鐘倍增到將近 14 分鐘。這促使我們從一個對平行處理支援有限的 CI 協調服務供應商，換到另一個提供更好原生平行處理支援的供應商。如果沒有足夠的可觀測性與長期歷史追蹤，我們根本無法在問題演變成開發人員極度挫折之前就察覺到它。追蹤建置時間的 SLO 讓我們得以避免長期的性能衰退——也就是「溫水煮青蛙」效應，開發人員感到挫折，卻只有軼事而沒有實際證據。

從 2020 年到 2025 年，我們每週合併到主分支的 pull request 數量從 10 到20 個，穩定成長到每週超過 100 個。這五年間，我們程式碼庫的總行數大約增加了一倍，使得編譯與單元測試所需的工作量都隨之增加。即使我們試圖增加平行處理，Amdahl 定律仍成為一個限制因素。

同樣地，我們能夠追蹤執行過程、找出緩慢的步驟以及「失敗後成功」的作業跨度，這讓我們得以判斷不穩定測試（flake）率，以及接下來該優化建置的哪個部分。我們需要找到正確的介入點，而不是進行不會影響整體關鍵路徑延遲的偏離目標的優化。

最終，我們到了無法再靠蠻力增加平行工作者或縮短建置步驟的地步。我們必須真正重新思考整個快取策略。我們是如何判斷自己已經無法再進一步的呢？Amdahl 定律指出，即使一個可平行化的步驟透過無限平行處理縮短到耗時為零，剩餘所有串行步驟的總和仍將主導關鍵路徑，也就是執行該工作負載所需的最短時間。

由於我們有來自 CircleCI buildevents 檢測的追蹤資料，我們得以研究這些資料，了解時間都花在哪裡。測試其實是建置過程中較快的部分，而二進位檔建置、產物建立與上傳步驟才是最慢的。為什麼會這樣呢？

建置步驟與資料傳輸

令我們驚訝的是，拖累性能的主要問題其實是在不同建置步驟之間傳輸檔案。近 3.5 GB 的資料在中間步驟與最終上傳建置產物的工作之間傳遞。我們發現這些資料是以 gzip 壓縮，而非 zstd，因此單靠壓縮無法有效縮減檔案大小。

與其設法壓縮資料，我（Liz）開始思考是否有可能完全避免搬移這些資料。我們需要送到 Kubernetes 叢集的主要產物是 Go 二進位檔，以及用 Webpack 打包並最小化後的 JavaScript。這些會被編譯進推送到 Amazon Elastic Container Registry（ECR）的映像檔，以及上傳到 S3 的未去除符號表（unstripped）二進位檔存檔副本。但隨著我們所有服務都轉移到容器與Kubernetes，已不再需要把去除符號表的二進位檔上傳成 tarball 存到 S3，也不需要把這些二進位檔納入 ECR 映像檔的一部分。如果我們能在單一步驟內完成所有建置、避免搬移狀態，會如何？

Docker bake

我們最初考慮的做法是用一個 Omnibus 建置步驟先做 Go 建置、再做JavaScript 建置，最後把兩者整合成建置產物。雖然這能將資料傳輸降到最低，但依序執行這些步驟——等前一步完成才開始下一步——的效率極差。我（Liz）的研究後來把我引向 Docker Bake，這是一種相對新穎的方法（2019 年推出，2022 年前後開始普及），用來串連多步驟的 Docker 映像檔建置。

平行建置

藉由讓各步驟能夠獨立進行——只在需要尚未完成的平行建置步驟輸出時才阻塞——我們得以在單一虛擬機上有效交錯執行多個平行建置。你可能會期待CircleCI 內建的 Docker 層快取（DLC）支援在此能派上用場。然而，我們的經驗是透過 CircleCI 工作區的物件儲存來傳遞狀態，速度比預期慢。用 DLC不加篩選地壓縮與解壓縮大量快取資料，所耗費的時間比從頭建立狀態，或是傳入從定期建置並明確快取的 tarball 產生的基礎層，都還要久。

消除瓶頸

把這些步驟從在 Docker executor 上執行的個別 CircleCI 工作，遷移到單一CircleCI machine executor 上的 Docker Bake，讓我們得以確立建置時間的下限。接著我們謹慎調整平行度，讓測試（仍在標準 CircleCI 平行執行器上執行，使用 CircleCI 平行測試分片

helper) 花費的時間大致相同。我們花了很多時間盯著使用 buildevents 檢測資料所產生的圖表，試圖理解每個步驟的延遲分佈，確保不會有單一工作成為瓶頸。例如，我們發現像是為 Go 整合測試初始化測試資料庫這類步驟——以前對總建置時間幾乎沒有影響——現在本身就需要進行優化。

快取網路擷取

我們也能夠深入個別追蹤，找出是否有特定步驟造成尾部延遲，並透過更積極的快取將這些延遲壓縮下來。舉例來說，我們發現用來更新 Puppeteer 系統依賴項的必要步驟通常執行得很快，但變異度可能非常大。因此，與其每次都重新下載，不如支付幾秒鐘的固定成本，快取當天第一次執行時的 apt 清單與.deb 套件。

成果

在調查過程中發生的一件真正很酷的事，是把 CircleCI 工作與步驟層級的buildevents 追蹤，與實際的 Docker BuildKit 指令追蹤資料串接在一起。事實證明，BuildKit 從一開始就整合了 OTel 支援，因此只要設定TRACEPARENT 環境變數以符合 buildevents 的追蹤與跨度 ID，我們就能直接觀察每個個別指令的執行、ECR 上傳步驟，以及等待其他步驟完成時的阻塞情形。

將建置時間持續維持在七分鐘以下的成果，絕對值得付出這番努力。隨著 AI產生的程式碼與拉取請求數量增加，這些好處也隨之加乘。能夠以語言模型產生程式碼的速度來驗證程式碼，是一項競爭優勢。

將這些經驗教訓應用到你的建置系統

既然你已經了解我們如何透過投資 CI/CD 流水線中的遙測與可觀測性來獲得工程生產力上的收益，接下來要說明如何從你自己的建置系統中取得類似的成果。

第一步，如本章前面所述，就是先在頂層了解情況有多糟。你需要知道每個建置步驟花費多少時間，即使你目前還無法把該建置步驟拆解成各個組成部分。畢竟，如果某個工作只花一秒鐘，而另一個阻擋建置流程推進的工作卻要花上兩個小時才能完成，那麼為前者的每個個別指令加入細粒度檢測可能根本沒有意義！

為了做到這一點，請使用你的建置系統原生支援的任何機制匯出資料。在Honeycomb 時，我們使用的是一款舊有的 Honeycomb 專屬工具，名叫buildevents；不過現在 OTel 是為新的建置工作流程進行檢測的最佳方式。舉例來說，GitHub Actions 可以透過兩種方式與 OTel 整合：一種是透過 OTel Collector 接收器，使用 GitHub API 金鑰拉取資料；另一種則是透過 GitHub Actions 附加元件，該元件會接收一個 OpenTelemetry Protocol（OTLP）目的地位址，用來發送有關目前執行中工作的遙測資料。而像 BuildKite 這類系統則原生支援匯出 OTel 追蹤資料。

若要追蹤個別的建置步驟，可以使用 otel‑cli 作為包裝工具。基本語法是執行otel‑cli exec ‑‑name "human‑readable span name" echo "do stuff"。接著，otel‑cli 會負責將追蹤狀態以及你設定的任何屬性序列化到環境變數（也可以選擇性地寫入磁碟上的檔案）。舉例來說，你可以在 otel‑cli 內部執行一個shell script 來做巢狀執行，並讓該 shell script 中的指令本身也用 otel‑cli 進行追蹤。如果建置步驟跨越了單一機器的邊界，你可能需要手動序列化該狀態並自行傳遞下去，而不是依賴 TRACEPARENT 環境變數或本地檔案系統。

但如果你的建置系統有部分或全部是用 Docker 撰寫，或是輸出結果本身就是容器映像檔，那該怎麼辦？考量到大多數雲端原生開發都部署在 Kubernetes上，因此需要在容器登錄檔（container registry）上建立容器映像檔，這並非可以忽視的問題。你的建置遙測資料中，會出現一個名為 docker build 或docker bake 的大型步驟，卻無法對其做更細緻的拆解。

與其試圖把 otel‑cli 導入 docker build 步驟並包裝每一道指令，不如利用Docker 本身原生支援的方式，直接在 docker client 中匯出 OTel 資料，或是透過 docker client 呼叫的底層 build 與 buildx 伺服器來匯出。以 docker client 而言，你可以在 CI 環境中、於 Docker 指令被呼叫之前，設定DOCKER_CLI_OTEL_EXPORTER_OTLP_ENDPOINT 這個變數。如果採用這種做法，就不應該同時使用 otel‑cli，否則會導致重複檢測。

請根據你目前的建置系統來調整檢測方式；優先使用最簡單的抽象化方法，只有在你的 CI 系統沒有內建、達到所需細緻度的自動檢測時，才對追蹤跨度進行手動檢測。而且，正如你會在正式環境系統中加入自訂屬性一樣，你也應該在這裡加入自訂屬性，用來追蹤輸出結果的大小、快取資料的數量、模組快取是否命中、使用已棄用或舊有模式的檔案數量等等，藉此從 CI 資料中榨取出更多價值。

結論

建置系統與可觀測性都是開發回饋循環的基礎。透過將建置系統視為一個生產系統，並使用相同的可觀測性工具來理解它，你就能消除軟體交付的瓶頸。CI/CD 系統可能既昂貴又脆弱，但可觀測性有助於你理解這些系統，並隨著時間降低風險。

在下一章中，你將學到如何把可觀測性應用到交付週期的另一端：行動裝置、網頁與前端系統的最後一哩路。
