可觀測性團隊的檢測工作

本章作者為 Hazel Weakly，Nivenly Foundation 研究員，以及 Austin

Parker，Honeycomb 的 AI 策略總監

拉爾夫・沃爾多・愛默生（Ralph Waldo Emerson）曾寫道：「語言是化石的詩歌。」1 社會學家與哲學家長久以來都觀察到，語言本身既塑造社會，也受社會所塑造。無論何處只要有兩個人以上聚在一起，語言就必然會發展或精煉出來。文化本身就是語言的熔爐——我們彼此互動、建立關係的方式，正是語言誕生之處。

一個正在為其系統擬定檢測策略的可觀測性工程團隊，面臨著類似的重擔。程式碼的老化就像一輛新車一開出展示間便開始貶值：它在被寫下的那一刻起就開始貶值。雖然軟體（整體而言）能夠帶來價值，但程式碼本身卻背負著維護與照料的重擔，需要謹慎的規劃、策略以及持續的整理，才能避免在系統擴展時讓維運人員不堪重負。如果這正是讓你夜不成眠的問題，那麼你來對地方了。

可觀測性團隊可能會因組織的規模、大小與職責範圍而有不同的樣貌。在較小的組織中，「可觀測性團隊」可能只是強加在站點可靠性工程師（SRE）身上的一個角色。較大的組織則可能有多個團隊，各自負責一組邏輯系統。你的上司可能隸屬於資訊科技部門，也可能隸屬於軟體工程部門。你的工作內容可能更像是供應商管理，而非其他。實務做法的範圍非常廣泛。

但整體而言，我們認為可觀測性團隊應該在相當廣泛的層面上擁有檢測的所有權。這包括檢測程式碼與框架、遙測

流水線配置與管理、警報基礎設施、SLO 工具等等。作為一個可觀測性團隊，你可能不負責親自撰寫檢測程式碼，但你很可能會被以其他人做了多少檢測工作來評估成效。

在本章中，我們將討論可觀測性團隊的檢測工作，包括不論團隊規模與型態為何，你的「北極星」應該是什麼。我們會先從檢測的基礎開始，介紹每個人都該知道的事。接著，我們會討論在安全、受監管或其他特殊環境中，檢測工作會有哪些不同。我們也會介紹在超大型企業中的可觀測性經驗與管理方式，最後討論可觀測性團隊應該如何看待基礎設施指標與可觀測性之間的關係。

可觀測性團隊需要知道的事

如前所述，語言是一種根據社會互動不斷演化的建構物。檢測也非常類似：它是從軟體不斷演化的運作方式中所產生的建構物。如果我們不需要透過日誌、指標、剖析（profiling）或其他技術，就能完美掌握程式碼在做什麼，那麼我們就不需要擔心檢測的問題了。

多年來，我們看過各種為軟體語意提出的空白石板（tabula rasa）出現。不幸的是，這些嘗試往往最終只適用於特定領域，泛用性不佳。舉例來說，.NET與 Microsoft Stack 長期以來享有執行環境與作業系統之間的垂直整合，讓Microsoft 得以產生一套或多或少一致的系統狀態元資料方言，但這套方言的泛化能力很差（後來 Microsoft 決定擁抱 Linux，也因此丟棄了其中一半）。

同樣地，Java 也有自己的方言與語意，貫穿整個技術堆疊——直到你加入React 前端或用 Swift 撰寫的行動應用程式為止。這時你又回到了原點。整體而言，*nix 系統擁有相對一致且自成體系的遙測方言，但如同大多數與 *nix有關的事物一樣，其學習曲線與其說是曲線，不如說更像是一道學習懸崖。

歸根究底，可觀測性團隊的北極星在於能夠為其組織定義並管理這套建構出來的語言。這需要一種能力：像考古學家一樣，梳理那些在公司軟體工程的非正式語言中逐漸固化的術語與定義，同時搭建一座通往外部世界通用語言的橋樑。作為可觀測性團隊的一員，你的角色不僅僅是分發檢測與文件而已，你還必須提供一套框架，讓橫跨不同團隊、事業單位乃至於整個公司的個別貢獻者與領導者，得以用來溝通意圖與行動。這是一項重大的責任與角色。

語意慣例

OpenTelemetry（OTel）對可觀測性團隊而言，最有價值的部分在於其語意慣例。語意慣例（有時口語簡稱 semconv）是一項協作成果，旨在為軟體、硬體與遙測本身的常見概念，定義標準化的人類可讀定義。

語意慣例的一個微妙目標，是讓遙測成為一項可訓練的技能，而不是必須在每個新職場中向資深開發人員耳提面命才能學到的東西。由於語意慣例在各種語言、雲端與執行環境之間是通用且可攜的，它們的作用比較像是一種方言，而不是一部百科全書。你不必刻意去背誦它們，但了解其結構的運作方式，有助於解讀不熟悉的慣例。

語意慣例是可觀測性中繼資料共通的詞彙與文法。詞彙指的是屬性、指標、跨度、事件等的命名集合，例如 http.request.method、db.system、exception.type、GET <route>。文法則是這些名稱如何組織成可重複使用的模式：以點分隔的命名空間、像 http.request.header.<key> 這樣的族群、必要性與成熟度等級，以及穩定性。

這樣的設計提供了可擴展性：你可以合理地從既有的領域形狀推斷出其他領域的形狀，讓探索與對應變得更容易。這也讓可觀測性成為一項更容易傳授的實務。工程師不需要記住每個服務個別的細節與繁瑣之處，一旦學會了基本的命名空間，就能輕易地追蹤服務呼叫鏈。大部分繁重的語意推理工作，其實只靠兩個經驗法則就能涵蓋：領域是由左到右依特定程度命名空間化，而行動者則是以客戶端／伺服器或生產者／消費者來定義。

語意慣例的另一項優點，是你不必事事親力親為。OTel 專案定義了為數眾多的領域，並提供了對應的檢測，同時還提供像 Weaver 這樣的工具，讓你能夠建立並定義自己的領域。對可觀測性團隊而言，領域定義與推廣正是檢測設計中最重要的環節之一，因此接下來讓我們聚焦於此。

實務中的遙測綱要

API 契約與版本控管的好處眾所周知：它們能提升部署的安全性，為使用端提供穩定的整合介面，並讓團隊更容易擴展規模。那麼，為什麼不用同樣的方式來對待遙測呢？遙測綱要這個概念，正是將現代 API 設計背後的最佳實務帶入可觀測性領域的做法，它透過文字形式來呈現服務所發出的遙測資料，並提供工具來探索、檢查、建立這些綱要，以及據以產生程式碼。

截至目前為止，導入遙測綱要（telemetry schema）並非一套現成即用的流程。但相關的基礎元件已經存在，可供你調整並實作。在缺乏逐步操作指南的情況下，讓我們來討論一下綱要的好處，以及如何撰寫新的綱要。

綱要為應用程式或服務所發出的遙測資料提供了單一事實來源。這對開發人員與 SRE 來說都是一大助益。用戶不必再面對五花八門的欄位與微妙重複的資料，而是可以參考一份與程式碼放在一起的單一文件。這種可發現性也帶來了各式各樣的應用：工具可以利用這些綱要，讓終端用戶更容易發現資料；稽核作業變得更容易；而 AI 助理也能將這些文件納入其上下文中，以進行更安全的編輯或代理式的 SRE 實務。

此外，這些綱要同時允許在編譯期與執行期對遙測資料進行驗證。當程式碼中出現未經指定的屬性或訊號時，建置可以被判定失敗或標示出來；也可以建立流水線，將產生的資料中的來路不明的屬性丟棄。

最後，綱要讓資料能夠安全地演進。流水線處理器可以利用綱要，在資料傳輸過程中更新或修改遙測資料，在資料被攝取並曝露給警報規則或儀表板之前先將其正規化。

撰寫新的綱要，是朝著在你的組織中將可觀測性這門建構語言正規化的北極星邁進的理想方式。以下提供幾個小訣竅：

從「為什麼」開始。

根據用戶可能會提出的問題來設計你的架構，同時也要支援足夠多樣化的欄位與屬性，以回答關於系統行為的突發性問題。讓常見問題的回答成本低廉、可靠且高效。

定義你的命名空間。

找出在整個組織中真正一致共用的業務屬性，並將它們置於命名空間中適當的層級。

DRY（不要重複自己）。共享的屬性鍵應該在不同信號類型之間共用。如果同一個屬性出現在不同

取而代之的是，你可以在信號之間分層屬性（見第 7 章），這樣既能提升你在不同信號間導覽的能力，同時也能限制基數。在這種情況下，你可能會想要新增一個 customer.region（或 tier 等）屬性，其基數較低，讓你能在時間序列上保留一些語意化的客戶中繼資料，而將較高基數的屬性保留給更適合它們的信號使用。

識別碼與安全團隊

在受監管的環境中，存在各種識別碼，這些識別碼一旦存在便會提高資料的合規等級（例如內部用戶識別碼）。這對可觀測性團隊來說是一項挑戰：安全性靠的是隔離，合規性靠的是證據，而可觀測性靠的則是關聯性。

在那些有用的識別碼會觸發合規或安全準則的情況下，你應該考慮對敏感資料進行單向雜湊處理。舉例來說，內部用戶識別碼可以用一種無法還原成原始格式的方式進行匿名化。只要整個組織採用的雜湊方法一致，你仍然能獲得關聯性帶來的好處，而不必承受安全與合規方面的監督負擔。

對遙測綱要（telemetry schema）的一項批評是，它們看起來是一種非常笨重的解決方案，尤其對較小型的組織而言更是如此。這是個合理的批評，但它反映的更多是組織內可觀測性團隊與開發團隊之間誘因不一致所產生的挫折與衝突，而不是這種做法本身固有的缺陷。

開發團隊常常沒有和可觀測性團隊一樣的顧慮，因為他們的關注範圍比較有限。他們可能只需要顧好單一服務，或是一小群服務，而可觀測性團隊卻要面對整個資料的消防水管（而且往往還要為控制這條水管的成本負責）。你也不太可能靠施壓其他團隊來讓他們正視這些顧慮。直接把成本記在帳上是一種做法，但這種做法通常會招來反感，也不會帶來良好的可觀測性成果。那麼，可觀測性團隊該如何打造正確的誘因，來鼓勵大家做出期望的行為呢？

工具、遷移與鋪好的道路

要讓開發、維運、業務與可觀測性團隊在遙測資料上取得共識，並不是發一份「最佳實踐」的 wiki 頁面就算了事，也不是隨便裝一套監控代理程式、丟給大家一堆現成的儀表板就行。

我們見過許多立意良善的 DevOps 團隊試圖「導入可觀測性」，結果花了好幾個月仔細分析每一項基礎設施與服務依賴關係，為那些服務推出代理程式設定與儀表板，把它們當成既成事實呈現給開發團隊，然後這些東西就這麼被束之高閣、蒙上灰塵，因為所收集的資料和上述任何一個團體的日常工作都毫無關聯。

請記住，你的北極星是為整個組織建立一套共通語言，並採用社會用來推廣語言的技巧：教育、辭典、儀式與重複（也就是文化）。用更實際的話來說，你需要打造工具與框架，讓用戶能夠輕鬆採用你正在建立的這套語言，為用戶在採用過程中建立可重複且容易取得的培訓機會，把檢測實務與策略的接觸點嵌入你的 SDLC 之中，並且將這些做法融入你的流程當中。

工具與框架

OTel 的設計旨在與既有的遙測系統、雲端環境和軟體開發生命週期（SDLC）工具具備可擴展性與互通性。在考慮如何建立自己的工具時，請牢記兩項基本原則：

1. 不要包裝（wrap）這個 API，但你可以擴展它，或在它周圍建立間接層。OTel 的設計目標是成為一個通用 API，具備通用的慣用語法。為了與既有遙測套件或個人偏好保持一致，而去包裝 API 並更改方法呼叫的名稱，是一種明顯的反模式。你可以擴展這個 API——例如撰寫

把多個呼叫捆綁成一個的裝飾器（想想同時啟動一個跨度並更新一組指標）——但要避免完全改變它。

你也可以選擇捆綁並重新匯出上游函式庫的方法。舉例來說，你可以建立一個單一的間接套件，將 OTel 與數十個現有的檢測 API 捆綁在一起。這麼做能讓你立即遷移許多服務而不會破壞呼叫端，同時在幕後重新接線。

2. 對用戶隱藏適當的複雜性。OTel SDK 可能非常冗長，帶有大量的樣板程式碼與客製化選項，感覺像座大山。這其中大部分的複雜性都可以濃縮成一行程式碼，加到服務框架中即可。目前這個領域的相關工作正致力於讓這件事變得更簡單，採用以設定檔為基礎的方法。

讓大多數開發團隊卡關的單一問題，就是服務之間的上下文傳播。解決這個問題正是一個絕佳機會，讓你能夠一開始就用打算長期維持的方式來做：把OTel 嵌入到你的遠端程序呼叫或服務框架中，搭配一份基本的遙測 schema，並確保這個框架能處理服務對服務之間的上下文傳播。如果採用 OTel 就像更新內部函式庫的版本一樣簡單，團隊就會更願意採用它。

在遙測產生的後端，你會想專注於管理與理解資料量與資料品質。OpenTelemetry Collector 以及像 Weaver 這類工具在這裡是至關重要的。Collector 具有極高的可設定性，同時也高度可客製化。與其嘗試在設定中處理所有事情，我們建議撰寫自訂擴充功能（只需要一點點 Go 程式碼），並使用 OpenTelemetry Collector Builder（OCB）打造你自己的 collector 發行版。這個選擇能讓你在處理獨特的過濾、取樣或轉換情境時擁有更大的彈性，同時也能降低資源使用量並縮小安全攻擊面。Weaver 可以整合進你的建置或部署流水線，即時檢查所發出的遙測資料是否符合已註冊的 schema，甚至可以加入作為本地建置／測試情境的一個 hook。

AI 在這裡也能成為助力。你可以探索對現有服務進行 AI 輔助修復：從預營運環境擷取遙測資料，讓它通過 Weaver 處理，然後把程式碼與 Weaver 的輸出交給一個 AI 代理，請它提出修復建議。如果你比較敢冒險（而且有完善的測試），你甚至可以讓代理自己動手去建立 PR。

對企業用戶來說，Weaver 還提供了額外的好處。確保遙測資料符合 schema雖然是良好的衛生習慣，但受監管的產業通常還需要對程式碼或輸出進行合規審查，以驗證沒有洩漏敏感資料。在 CI 中使用 Weaver 來為合規團隊與稽核人員蒐集證據，能將這個繁瑣的過程自動化。

鋪好的路徑與特殊帽子

我們經常看到，OTel（以及廣義的可觀測性）計畫遭遇用戶令人咋舌的冷漠反應。你可以帶一位開發者到水邊，卻不能強迫他喝水。這其實一點也不令人意外。即使在最好的情況下，可觀測性都可能讓人感覺是件苦差事，也難怪整個產業似乎傾向於追求「一鍵搞定」的按鈕，以及承諾能替你完成所有工作的神奇代理程式（或者，現在則是承諾能替你理解並解決問題的神奇 AI 代理程式）。

一個可能相當具說服力的故事，來自 Google 分散式追蹤發展初期。當時負責Dapper（Google 內部追蹤工具）的其中一位工程師，正苦思該如何爭取更多用戶。Dapper 早已推出，也持續從各種服務（搜尋、購物等等）收集追蹤資料。但那種「叫大家去用它」的游擊式行銷方式，卻停滯不前。

這位工程師的導師建議，先弄清楚大家原本就已經在哪裡：他們平常是用什麼工具、畫面和服務來理解性能表現的？只要搞清楚這一點，路自然就會浮現。

Dapper 團隊訪談並分析了工程師的行為，最終在團隊被呼叫（page）時經常查閱的監控圖表上，加了一個連結，直接導向 Dapper。使用量幾乎在一夜之間翻倍，接著又翻了三倍。發現 Dapper 的工程師會告訴其他團隊的同事。不久之後，分散式追蹤便成了 Google 營運環境監控與可觀測性中的核心工具。

你的工程團隊很可能有類似的聚集地：現有的儀表板、警報、wiki 頁面等等。找出這些地方，理解它們的痛點，鋪出一條通往你的可觀測性資料與 OTel 解決方案的道路，然後讓奇蹟發生。你不會一下子就贏得所有人的認同，但總得從某處開始。

如前面所提到的，透過建立工具讓採用變得盡可能容易。前期的一點投入會隨著時間帶來回報，尤其是對那些不太配合的團隊而言。「正確的方式」和「最快的方式」應該是同一種方式。我們在成功的組織採用案例中看到的一種做法，是彙整並宣傳 OTel 的成功案例——保留一本「戰功錄」，記錄可觀測性工作帶來幫助的地方，並確保管理層看得到。做一些貼紙、T 恤或小徽章來激勵大家採用與使用。發揮創意吧！

最後，讓採用變得安全。運用 Collector 流水線將資料同時傳送到便宜的 blob儲存以及遙測資料庫——尤其是在進行遷移或變更取樣規則時——藉此比較新舊資料，並確信自己沒有遺漏任何東西。加入規則（透過 transform processor、schema 或其他機制）以避免基數爆炸，方法是針對指標等特定訊號類型丟棄高基數欄位。為本地環境提供預設流水線

開發階段就設計為易於讓開發者理解正在發送哪些遙測資料，並集中管理流水線設定，以便清楚了解為何某些資料會被丟棄。

採用方面的最後一個提示對取樣尤其重要。開發者最感挫折的時刻之一，就是他們去尋找原本以為存在的資料，卻發現它不存在。雙重發送（dual‑send）可以在資料被取樣排除時提供備援，藉此緩解這個問題。但你也可以透過加強教育來幫助大家，讓大家更了解取樣、它的優點與缺點。務必讓取樣率易於理解，並在一開始避免過於複雜的取樣策略。一般來說，剛開始時寧可取得「過多」的資料，之後隨著你對工具建立起信心，再逐步縮減會比較好。

總結來說，在推動檢測與 OTel 的過程中，把自己想成是一位可觀測性團隊的嚮導會很有幫助。你不見得是那個站在山頂拍照的人，但你會很努力確保跟你一起上路的人沒有因此送命。 OTel 給了你一個框架，讓你能建立安全、可靠且可重複的檢測路徑——而且隨著它逐漸成熟，這件事只會越做越容易。它提供了你和你的組織所需的語法與詞彙，作為你在其上搭建自身概念的基礎。它提供工具來協助管理並限制現代雲端原生軟體所產生的遙測量，並且只要善用其提供的功能，就能幫助你避開常見的陷阱。學習它、善用它，甚至喜歡上它，但不要輕視它——它是一波你可以乘著加速你組織採用可觀測性的浪潮。

在高度安全環境中的檢測

在高度安全的環境中，關於檢測的兩大主要考量是合規性與安全性。雖然這兩者都是極為複雜的主題，但從可觀測性的角度來看，我們將合規性視為主要與所提供的證據有關，而安全性則主要與所保證的隔離性有關。

不幸的是，這意味著我們很快就會遇到一個棘手的情況：可觀測性的超能力在於資料關聯。若做得不夠謹慎，這項超能力很容易破壞安全隔離的保證，並阻礙合規證據的蒐集。然而，如果我們將可觀測性視為整個企業共通的語言，就有可能在不損及安全性的前提下促成合規。

考量到這兩項顧慮讓實作可觀測性變得困難許多，或許令人意外的是，這與在任何企業中實作可觀測性其實仍非常相似。其中一項主要差異在於實作安全能力的門檻有多高；許多「最佳實務」都變成了硬性要求。

另一個顯著的差異在於舉證責任有多沉重。你有端對端加密嗎？你有妥善保護客戶資料嗎？你的 OTel Collector 是否使用 FIPS 140‑2 加密？很好！你將需要為以上所有事項提供證據。此外，要在這方面取得成功，很大程度上取決於能否與各團隊妥善溝通並設計合適的誘因：領導層、工程、產品、架構以及其他每個職能部門都必須有共識。

如果你覺得說服利益相關者與你合作實作可觀測性已經很困難，那麼在一個合規相關瑣事多出十倍的組織裡說服他們，又是另一回事了。

許多處於這種情況的組織會陷入一個陷阱：只能在「強制性」項目上取得進展；不幸的是，可觀測性幾乎不可能被列為強制性項目。要在這裡取得成功，通常需要以某種方式設計誘因，讓每個參與者在協助你的時候都能顯得有貢獻；他們通常會想要可觀測性，但需要在一堆強制性工作面前為此提出正當理由。幫助他們做到這一點。

這也是為什麼談論以可觀測性協助促進合規證據蒐集會顯得合理的原因之一。2

善用合規分層

幸運的是，高度安全環境還有另一個要素，能讓可觀測性更容易實作：極少有環境需要每一項合規控制。

沒有人希望為自己增加不必要的工作，因此常會遇到多層合規要求、多層法規，以及採取漸進式方式來管理不同環境。善用這一點，提供順暢的方式讓服務從一種合規等級「升級」到另一種等級。新創公司可能會對花 12 個月才能讓應用程式「進入營運環境」的想法感到卻步，但這在受監管的產業中卻是常態。

話雖如此，服務不「在營運環境」並不代表團隊沒有在部署程式碼。服務通常會先在本機環境建置，接著依序經過開發、測試、預備（staging），最後才是營運環境。重要的是，每個層級的要求都更加嚴格。常見的錯誤是提供一套「營運環境就緒」的可觀測性框架，把實作負擔轉嫁給開發團隊。

與其要求開發團隊耗費數個月的瑣碎工作，才能在本機環境「取得一些資料」，不如讓他們能夠在既有條件下盡可能快速地進行實驗

每個環境的限制。你會驚訝地發現，當你努力減輕他們的工作負擔時，他們很快就會成為你的擁護者。不要只是把可觀測性工具往左移，也要把價值往左移。

說到把價值往左移，這通常意味著以平台的形式提供一組最小化的能力，將可觀測性整合進公司的需求之中。

可觀測性團隊所需的最小能力集

如果我們要協助處理合規性與安全性方面的問題，以下是值得思考的能力起點：

供應商管理
你的監管機構會對你提出許多要求。他們會想知道你的供應商是否有能力，而你也需要一些強而有力的商業協議與財務誘因，確保你與供應商之間的關係是互利的。要有心理準備，為你所要求的東西付出代價。

加密（端到端、靜態資料等）

作為一項要求，端到端加密的徹底程度可能超乎你的預期。你需要在每一個元件之間都採用 mTLS，即使是位於同一個信任邊界內的服務也不例外。只要是透過網路通訊，就必須使用 mTLS，沒有例外。這也是為什麼在可能的情況下，服務間通訊應優先使用 socket 的重要理由。

雖然 mTLS 處理了大部分的端到端加密，但對靜態資料進行加密同樣重要。視你的需求而定，加密硬碟可能已經足夠，也可能不夠。

多層次資料分析
這是最困難的一環。你必須證明，你透過網路傳送到任何地方的每一個位元組資料，都符合所有監管要求。光是符合要求還不夠，你還必須展示出你是如何做到的。

舉例來說，許多資料法規要求，只有經過客戶明確授權的最小相關資料，才能被傳送到指定地點。你必須非常小心，避免在遙測資料中不小心傳送出可識別身份的資料。可觀測性的超能力在這裡反而會對你不利：關聯性正是可觀測性的一大優勢，但這也意味著，原本在局部看來「安全」的遙測資料，很容易在全域層面被拼湊組合成去匿名化、可識別身份的資料。

你會注意到一個模式：上述所有能力都是不可見的。不會有任何面向客戶的產品藍圖會優先開發這些能力，就連可觀測性團隊自己在實作這些能力時，也很難展現其價值。因此，能夠盡可能快速且高效地實作這些能力，就變得至關重要。

監管環境中的重要能力

正因如此，以下能力在監管環境中運作時變得相當重要，若缺少它們，可觀測性幾乎難以實現：

成本回沖（Chargeback）這對於確保可觀測性團隊擁有適當的人力配置、並在公司內展現其價值非常有用。若沒有這一點，可觀測性工作可能會因缺乏支持而逐漸消失。

這也可以被視為將成本與運營考量納入企業的「語言」之中。商業智慧形塑的考量往往與監管要求密切相關，尤其是那些涉及「過度負擔」用語的法規。

架構描述控制（Schema control）沒有這項能力，你依然可以進行多層次的資料分析，但當能夠自動針對架構描述（schema）進行檢查時，要擁有多層級的機密資料就會容易得多。架構描述也能更輕鬆地防止全域關聯性破壞客戶資料的監管匿名性。換句話說，明智地運用 OTel Weaver 專案，可能是決定你是否需要為每個應用程式（而非每個環境）各自部署一套可觀測性系統的關鍵差異。它甚至可能決定一個團隊是否被允許取得除錯資訊。

透過你所掌控的代理伺服器傳輸資料

嚴格來說你不一定需要這項能力，但在幾乎所有監管環境中，為了合規與安全理由對資料進行運行時掃描都是必要的。不透過代理伺服器來做也是可行的，但會徒增不必要的痛苦。

足夠強大的代理伺服器，是讓政策代理程式（policy agents）成為有效工具、以強制執行全公司資料控管的關鍵。它們對於為那些原本難以證明的要求（例如允許傳送給第三方供應商的資料範圍）提供證據，也格外有用。

代碼化與資料匿名化（Tokenization and data anonymization）代碼化能讓你做到一件非常強大的事：它可以協助防止「在本地安全的資料」被「全域關聯」起來。如果資料因為已加密而無法被匿名化，你就不必以同樣的方式擔心身份識別的問題。

代碼化還有另一項強大的好處：它讓你能夠在你的工具之外進行側通道關聯分析。那些原本可能過於敏感或過於冗長、不適合放進可觀測性資料中的內容——例如 AI 對話紀錄或資料庫查詢——透過代碼化這類技術結合使用後，就能變得有用。

如前所述，匿名化識別碼的關聯性帶來一些好處。使用安全、單向、且在整個組織中保持一致的雜湊函式，可以在將資料曝險降到最低的同時，提供一定程度的功能性。

授權與驗證

你的資料處理作業需要確保是由被允許執行此作業的代理人所處理。如果你的 mTLS 採用短期憑證、驗證過的身分（attested identity）以及細緻的權限控管，將能更輕鬆達成這一點。

此外，許多法規要求涉及經營技術組織的「非技術」層面。你將會遇到需要記錄並執行長期支援計畫的需求，也需要為遷移作業制定運作模型。最佳實務指南會演變為藍圖與框架。文件記錄則會進一步發展為授課與內部支援 SLA 的要求。程式碼掃描與 schema 驗證，也需要在開發階段、CI 中、部署期間，以及應用程式運行期的整個生命週期中執行。

這是艱難的工作，但值得投入

身為可觀測性團隊，思考所有這些法規要求可能會讓人感到不知所措，甚至覺得自己所做的安全與合規工作是在「額外附加」於可觀測性所帶來的價值之外，或是「取代」了那些價值。然而，主動與安全和合規團隊合作，將在未來建立信任、掃除工程團隊面臨的障礙上帶來豐厚回報。

畢竟，如果工程團隊無法有效地對其程式碼進行檢測，那麼可觀測性的其他一切最終都將無關緊要。

就像人類的口語一樣，當每個人都站在同一個立場、彼此有效溝通時，可觀測性的力量才能真正展現出來。以安全、可靠且合規的方式，賦予他們這種語言以及表達它的工具，正是在高度安全的環境中交付現代工程能力的祕訣。

結論

可觀測性團隊的角色終究不是去宣稱可觀測性計畫的成功，而是要為整個組織促成這樣的成功。這使得定義成功變得更加困難；相較之下，資料量與儀表板數量要容易衡量得多。

你的成果會以更微妙的方式呈現。原本要開一週電話會議才能解決的問題，現在幾天就能收斂。團隊用服務水準目標（SLO）來溝通期望，主管與高層也會呼應這套語言。合規不再是個令人頭痛的難題。不過請記得，現在很可能是開始建立可觀測性計畫的最佳時機。
